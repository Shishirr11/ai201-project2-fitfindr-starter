# FitFindr

FitFindr is an AI agent designed to help users thrift and style secondhand clothing. It takes a natural language request, searches a mock database for matching items, styles the found item with the user's existing wardrobe, and generates a shareable social media caption.


## Tool Inventory

### 1. `search_listings`
* **Purpose:** Searches the mock listings JSON dataset for secondhand items that match the user's specific criteria.
* **Inputs:** * `description (str)`: The style, brand, or keywords of the item being searched for.
  * `size (str)`: The specific clothing size requested.
  * `max_price (float)`: The absolute maximum price the user is willing to spend.
* **Outputs:** `list[dict]` - A list of dictionaries representing matching listings.

### 2. `suggest_outfit`
* **Purpose:** Analyzes the newly retrieved item alongside the user's existing wardrobe to generate personalized styling advice.
* **Inputs:** * `new_item (dict)`: The dictionary of the single clothing item selected from the search results.
  * `wardrobe (dict)`: The dictionary representing the user's current closet contents.
* **Outputs:** `str` - A text string containing actionable styling advice.

### 3. `create_fit_card`
* **Purpose:** Translates the detailed styling advice into a short, punchy, shareable description suited for a social media caption.
* **Inputs:**
  * `outfit (str)`: The styling paragraph generated by the `suggest_outfit` tool.
  * `new_item (dict)`: The dictionary of the selected item to pull in details like price and brand.
* **Outputs:** `str` - A short text string formatted like an Instagram or TikTok caption.

## Planning Loop
The agent uses sequential, conditional logic based entirely on the output of the first tool. It always initiates by running `search_listings`. It then checks the length of the `results` list. **If the list is empty**, the agent explicitly halts execution, sets an error string in the session, and returns immediately without calling the next two tools. **If the list is populated**, it assigns `results[0]` to the session state and proceeds to call `suggest_outfit`, followed immediately by `create_fit_card`. It knows it is done when the fit card string is successfully generated and saved to the session.

## State Management
Information is passed using a central `session` dictionary that persists throughout the run. When `search_listings` succeeds, the top item is saved to `session["selected_item"]`. The next tool, `suggest_outfit`, pulls its inputs directly from `session["selected_item"]` and `session["wardrobe"]`. The resulting advice is saved to `session["outfit_suggestion"]`, which the final tool then reads to generate the caption.

## Error Handling Strategy
Every tool is designed to fail gracefully:
* **`search_listings`:** If a user searches for an impossible item (like testing with "designer ballgown size XXS under $5"), the tool returns an empty list `[]`. The agent catches this, stops the loop, and outputs: *"I couldn't find anything matching those filters. Try adjusting the size or price!"*
* **`suggest_outfit`:** If the user's wardrobe file is empty, the LLM prompt defaults to suggesting universal style staples (like plain jeans) to ensure an outfit is still generated.
* **`create_fit_card`:** If the outfit string is missing, the tool falls back to a hardcoded caption that skips the styling advice and just highlights the item's price and brand.

## Spec Reflection
Writing the `planning.md` specification before coding was incredibly helpful for visualizing the error paths and defining exact failure behaviors. Having the fallback strings pre-written meant I wasn't trying to design user experience and debug API calls simultaneously. One divergence from my spec was how I parsed the user's query; I originally intended to rely on the LLM or simple splits, but I pivoted to using regular expressions (`re.search`) to explicitly extract the price and size, which made the agent much more robust and faster.
